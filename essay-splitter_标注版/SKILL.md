# 申论讲义拆分与 PDF 转换

拆分一份含多讲、多题目的申论讲义 Markdown，生成「一个题目一个文件」的独立 MD，再批量渲染成 PDF（PDF 放到独立的 `PDF/` 子目录）。

## 核心原则（踩过的坑）

1. **题目识别不能只看「真题/试题」**。申论讲义里题目块的三级标题至少有这几种：`### 题目`、`### 真题 1/2`、`### 试题 1/2/3`，**都要覆盖**。否则第1讲、第5讲、第7讲这种「一讲一题」的情况会被整讲漏掉。
2. **共享材料要合并**，不能把共享材料的题目各拆一份（否则同一段材料会重复抽出，且最后一题会只剩题干没材料）。
3. **文章写作（大作文）讲结构特殊**：用 `#### 给定资料X` 标记多段材料，可能还出现 `同第X讲题目材料` 这种跨讲引用，题干在最后。必须把所有给定资料段**按顺序拼接**作为材料，跨讲引用要去源讲抓回来。
4. **不要漏讲**。先打印所有「讲」和所有「题目候选行」给用户对齐，确认每讲应输出几个文件，再动手。

## 第一步：读源文件

```python
import sys
sys.stdout.reconfigure(encoding='utf-8')

with open(source_path, 'r', encoding='utf-8') as f:
    lines = f.read().split('\n')
```

必须 `encoding='utf-8'`，Windows 下不指定会乱码。

## 第二步：扫描结构（全量、分类）

用一次扫描收集 4 类标记的行号：

```python
import re

lectures = []       # ## 第X讲
subheaders = []     # ### 真题/试题/题目 （这一级是"材料块起点"）
questions = []      # **根据…** 或 **请根据…** （题干）
share_markers = []  # 同试题X材料 / 同上材料 / 同第X讲题目材料
data_headers = []   # #### 给定资料X / #### 给定材料X （大作文用）

for i, line in enumerate(lines, start=1):
    s = line.strip()
    if re.match(r'^##\s*第\s*[\d\-\\]+\s*讲', s):
        lectures.append((i, s))
    if re.match(r'^###\s*(真题|试题|题目)', s):
        subheaders.append((i, s))
    if s.startswith('**') and ('根据' in s or '结合' in s):
        questions.append((i, s))
    if re.search(r'同(试题|上|第)[^，。\s]*材料', s):
        share_markers.append((i, s))
    if re.match(r'^####\s*给定(资料|材料)', s):
        data_headers.append((i, s))
```

**先把这 5 类标记全打印给用户看一遍**，让用户确认「这份讲义应该拆成 N 个文件」，否则容易漏。

## 第三步：决定文件列表（最关键）

对每个「讲」，逐一判断产出几个文件。常见 4 种模式：

### 模式 A — 一讲一题（`### 题目` 型）
讲里只有 1 个 `### 题目`（或只有 1 个真题/试题），该讲下只有 1 道题。
→ **1 个文件**。材料 = 题目块起始到题干前一行的所有非标题内容。

典型：第 1、4、5、7 讲。

### 模式 B — 一讲多题、材料独立
讲里有「真题1」「真题2」（或「试题1」「试题2」），每个后面各有自己独立的一段材料。
→ **每个真题 1 个文件**。

典型：第 2 讲（X县 / D县）、第 6 讲（Q县 / 乡镇）、第 8 讲（文化大院 / 基层党建）。

### 模式 C — 一讲多题、材料共享
讲里有多个子题，但后面的子题附近出现「同试题1材料」「同上材料」之类的标记。
→ **整讲合并为 1 个文件**。一份材料 + 所有题干（题干之间用空行或 `### 试题X` 小标题分隔保留）。

典型：第 3 讲（冠军企业 3 个子题共享材料）。**不要把「启示」那一题再拆出来**。

合并文件的**题型名按第一道题的题型写**（即使后面子题题型不同也不拼接）。例如第 3 讲 3 个子题中试题1是归纳概括、试题3是综合分析，文件名题型统一写「归纳概括」，不要写成「归纳概括+综合分析」。

### 模式 D — 大作文讲（多段 `#### 给定资料X`）
讲里有若干 `#### 给定资料X` / `#### 给定材料X` 段，最后一个 `**请…撰写一篇文章**` 是题干。其中某段给定资料可能写着「同第X讲题目材料」，需要回到第 X 讲把材料抓过来塞进去。
→ **1 个文件**（议论文），材料 = 所有给定资料段按原顺序拼接（遇到跨讲引用就替换成被引用的材料正文）。

典型：第 9-10 讲（常青之道贵在自胜）。

### 判断流程

```
对每个讲：
  subs = 该讲内 subheaders 数量
  data = 该讲内 data_headers 数量

  if data >= 2:
      → 模式 D（大作文），输出 1 个文件
  elif subs <= 1:
      → 模式 A，输出 1 个文件
  else:
      if 该讲内出现 share_markers:
          → 模式 C，输出 1 个文件（合并）
      else:
          → 模式 B，输出 subs 个文件
```

**写完判断后，一定把"每讲 → N 个文件"的结果整个列出来给用户确认**。对着用户给的目标清单数个数，漏一个就返工。

## 第四步：提取内容

### 通用规则

- **材料起点**：从该题目块标题的**下一行**开始，跳过空行和所有 `#` 开头行，找到第一个实际内容行。
- **材料终点**：题干行的前一行。
- **特殊情况：小节下没有独立的新材料段（如第 6 讲·真题2）**——这一小节的 `### 真题X` 标题下紧接着是正文叙述、再跟题干，没有再出现一个明显的新材料分段。此时**材料 = 前一题的题干结束位置（或前一个 `---` / `### 小节标题`）到本题题干之间的所有正文**，也就是「与前一题之间的间隔」整段都算本题材料，不要往上越界到前一题的材料区。
- **题干**：从 `**根据…**` / `**请根据…**` / `**请结合…**` 行开始，往下收集「要求：…」「不超过X字」等配套行，遇到 `---` 或下一个 `#` 开头行或空白分隔符停止。

### 模式 C 的特殊处理（共享材料）

1. 材料只抽第一个子题块下面的那段（后续子题下面除了「同试题1材料」标记外通常没内容或只有题干）。
2. 所有子题的题干按顺序拼接，中间用 `\n\n` 空行分隔，保留 `### 试题1` / `### 试题2` 这类小标题帮助阅读。

### 模式 D 的特殊处理（大作文）

1. 遍历该讲内所有 `#### 给定资料X` 段：
   - 如果这段正文是「同第X讲题目材料」这种引用行，**回到第 X 讲，默认自动把那一讲的材料（即模式 A 下抽取的材料）原文塞进来**，加一行 `#### 给定资料X（引自第X讲）` 作为小标题。这是默认行为，不需要逐次向用户确认。
   - 否则保留该段所有非标题的实际内容，标题 `#### 给定资料X` 保留（帮助阅读）。
2. 最后追加题干（`**请结合全部给定资料…**`）和"要求"列表。

## 第五步：文件命名

格式：`{原文档大标题}_{两位数序号}_{题型}_{主题}.md`

- **原文档大标题**：源文件首个 `# 标题` 的文本（如 `2025山东省考申论系统班讲义`）
- **序号**：两位数 `01`、`02`…… **按全文档题目出现顺序连续编号，不按讲分段**
- **题型**：归纳概括 / 提出对策 / 综合分析 / 应用文 / 议论文（模式 D 一律写"议论文"）
- **主题**：5-10 字题目内容概括

示例（完整 12 个）：

```

├── 2025山东省考申论系统班讲义_02_…….md
```

**PDF 统一放在根目录下的 `PDF/` 子目录**，不要和 MD 混在一起。目录不存在就创建。

## 第七步：批量转 PDF

**不要用 pandoc**——中文支持差。用 Playwright：

```bash
pip install playwright markdown
playwright install chromium
```

调用 `scripts/md_to_pdf.py`：

```bash
python scripts/md_to_pdf.py <md输出目录> <md输出目录>/PDF
```

脚本按系统中文字体（Microsoft YaHei / SimHei / PingFang SC）渲染，输出与 MD 同名的 PDF。

## 与用户对齐的三个强制确认点

开工前、关键分叉点停下来找用户对齐：

1. **源文件路径与输出目录**
2. **扫描结果**：打印所有讲标题、题目候选、共享标记、`#### 给定资料X` 的行号和内容，让用户过一遍
3. **文件清单**：在动手抽取前，列出"我准备生成这 N 个文件，序号 / 题型 / 主题"清单，等用户确认 OK 再跑

哪怕判断逻辑再准，最后生成前过一遍这三步，能避免漏题、重复、错归类。

## 第八步：生成带踩点标注的精读版（标注版）

拆完基础 MD 后，**默认再给每个题目生成一份"标注版"**，用于学习和踩点训练。规则如下：

### 输出位置

根目录下新建 `标注md/` 子目录，每个基础 MD 对应一份标注版，文件名加 `标注版_` 前缀：

```
输出根目录/
├── 2025山东省考申论系统班讲义_01_归纳概括_济钢无钢发展经验.md
├── ……
├── PDF/
└── 标注md/
    ├── 标注版_2025山东省考申论系统班讲义_01_归纳概括_济钢无钢发展经验.md
    ├── ……
```

### 两种标注元素

标注直接嵌入原文正文，**不要改写原文、不要删原文**，只做增量标注。

| 元素 | 语法 | 用途 |
|---|---|---|
| 加粗 | `**xxx**` | 标示可直接作为答案要点的关键信息——材料原词、核心关键词、核心动词、具体数据、专有名词、政策名词 |
| 方括号注解 | `【xxx】` | 紧跟在被标注对象后面，用**行内代码样式 `【xxx】`** 或直接的 `【xxx】` 文字，说明要点类型、踩点提示、易错警示、非得分背景等 |

### 标注内容的 7 类功能（用户参考分类）

每条 `【…】` 注解应属于下列其中一类（开头用类别关键词点明）：

1. **核心观点/主旨升华**：`【核心观点句：xxx】`、`【结尾展望：xxx】`
2. **问题剖析**：`【核心问题一：xxx】`、`【问题表现①：xxx】`、`【问题根源：xxx】`
3. **措施/对策/成效**：`【⭐核心经验1：xxx】`、`【具体措施：xxx】`、`【对策方向：xxx】`、`【效果：xxx】`
4. **概念界定/本质分析**：`【本质揭示：xxx】`、`【定义收束：xxx】`
5. **背景/引述（非得分点）**：`【背景：xxx，非直接得分点】`、`【场景引入：xxx】`、`【案例：xxx】`
6. **评分规则/答题规范**：`【2分】`、`【1分】`、`【易错警示：xxx】`、`【答题提示：xxx】`、`【必须使用原词，不可替换为xxx】`
7. **成效反馈/群众评价**：`【直接成效：xxx，得分关键词】`、`【群众感受原文：xxx】`

### 标注操作要点

- **原词加粗优先**：能作为答案得分点的实词（动词、名词短语、数据、专有名词）都要加粗。
- **加粗 + 注解组合**：关键句通常先加粗关键词，再紧跟 `【…】` 说明它为什么重要。
  例如：已陆续关停 **50 余家**【具体数据+核心措施动词，行政手段，直接得分点】清河沿岸的 **"散乱污"小作坊**【关停对象，专有名词】。
- **非得分背景必须标**：凡是铺垫、过渡、案例引入、时间节点这类不能写进答案的句子，要加 `【背景：xxx，非直接得分点】` 提醒不要误写。
- **易错点必须标**：材料中容易被误概括的地方，加 `【易错警示：xxx】` 或 `【必须使用原词，不可替换为xxx】`。
- **分值提示**：如果题目能推断出采分结构，每个要点后面加 `【2分】`、`【1分】`。
- **⭐符号**：核心经验/核心对策/核心观点句前加 ⭐，便于快速扫读。
- **题干也标**：`**根据给定资料……**` 这一行，把"归纳概括/请结合/谈谈认识"等题型动词加粗，并注明题型与字数要求。

### 标注示例（对照）

原文：
> 2020 年年底，北溪县基本实现了县到乡、乡到行政村、行政村到村民小组 100% 通水泥路的目标。为了准确掌握全县农村公路……

标注版：
> **2020 年年底**，北溪县基本实现了县到乡、乡到行政村、行政村到村民小组 **100% 通水泥路**的目标。【建设成效背景，非直接得分点】为了准确掌握全县农村公路建、管、护、运营的实际情况，制定出切实可行的 **农村公路提升工作方案**，由 **县交通运输局、财政局、农业农村局、乡村振兴局** 等组成的 **联合调研组**，于 2021 年 9 月开始对全县乡镇的农村公路进行了为期一个月的实地调研。【背景：调研组开展调研，引出下文问题】

### 标注版也要转 PDF

标注版同样要渲染成 PDF，放到 `PDF/标注版/` 子目录下。`【】` 在 PDF 里用灰色底、小号等宽/楷体等不同字体呈现，以便视觉区分。可以在 md_to_pdf 的 HTML 模板里加一条 CSS：

```css
/* 【括号注解】专用样式 */
body { font-family: "Microsoft YaHei", sans-serif; }
/* 用正则把文中的【…】包成 <span class="annot"> 再渲染 */
.annot {
  font-family: "KaiTi", "楷体", monospace;
  font-size: 10pt;
  background: #f2f2f2;
  color: #666;
  padding: 1px 4px;
  border-radius: 3px;
  margin: 0 2px;
}
```

在 md→html 转换后、传给 Playwright 前，用正则 `【([^】]+)】` → `<span class="annot">【\1】</span>` 做一次替换。

### 流程上的位置

完整流程变成：
1. 拆分生成基础 MD（到根目录）
2. 渲染基础 PDF（到 `PDF/`）
3. **逐个 MD 生成标注版**（到 `标注md/`，加 `标注版_` 前缀）
4. **渲染标注版 PDF**（到 `PDF/标注版/`）

第 3 步是 Claude 阅读材料、理解要点、手动打标注的过程——不是机械正则能完成的，需要真正读懂材料，判断哪些是答案点、哪些是背景、哪些是易错点。做之前可以先问用户 1-2 个已有的标注样例作为风格参考。

## 常见坑汇总

- **编码**：Python 读写都要 `encoding='utf-8'`；Windows 终端打印中文加 `sys.stdout.reconfigure(encoding='utf-8')`
- **路径**：Windows 路径用原始字符串 `r'C:\...'` 或正斜杠
- **`### 题目` 不要漏**：和「真题/试题」一样都是题目块起点
- **共享材料题不要重复拆**：出现「同X材料」就整讲合并，不要再给最后一题单独来一份
- **大作文的 `同第X讲题目材料`**：要回溯到被引用的讲抓材料回来
- **题目序号连续**：跨讲连续编号，不是每讲从 01 重新开始
- **PDF 目录隔离**：单独放 `PDF/` 子目录
- **pandoc 不行**：中文 PDF 用 Playwright
